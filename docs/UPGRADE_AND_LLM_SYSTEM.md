# 升级与 LLM 批改系统说明

本文说明从**未配置或部分配置 LLM** 的旧系统升级到新版本时的注意点，以及作业自动批改相关行为，便于运维与教务排查。

## 1. 升级能否在「未配好 LLM」的情况下完成

**可以。** 核心业务（用户、班级、课程、作业布置与提交、教师人工评分等）不依赖 LLM 端点即可运行。

- 应用启动时会执行 **`ensure_schema_updates()`** 等迁移逻辑，为数据库**增量添加**新列与新表（例如作业批改队列、token 用量表、作业 `max_submissions` 等）。旧数据缺省列会使用数据库默认值（如 `NULL` 表示「不限制提交次数」）。
- 作业字段 **`auto_grading_enabled`** 默认为关闭；未开启自动批改时，学生提交**不会**要求可用的 LLM 配置。
- **批改 Worker** 可通过环境变量关闭（见 `tests/conftest.py` 与生产配置中的 `ENABLE_LLM_GRADING_WORKER` 等），避免在未就绪时占用进程或反复失败。

**建议在升级后检查：**

1. 若暂不使用自动批改：确认历史作业与新作业均未误开「自动评分」，或接受「已开启但尚未配置端点」的作业在队列中失败并在界面显示错误信息。
2. 若计划使用自动批改：在管理端完成 **LLM 端点预设** 与校验，再在**课程 LLM 配置**中绑定端点（仅可选择已通过校验的预设），并确保 **Worker 进程**在生产环境中已启用且能访问外网模型 API。
3. 数据库备份与回滚策略仍按常规发布流程执行；本文不替代正式运维手册。

## 2. 自动批改与 Token 预算的关系

- 自动批改是否触发，首先取决于作业是否 **`auto_grading_enabled`**，以及提交后是否成功 **入队**；之后还受课程 LLM 开关、端点配置、Worker 是否运行、模型可用性等影响。
- **每人每日输入 token 上限**由管理员在全局策略中配置默认值，并可对个别学生做覆盖；该数值是「每个学生每天」的上限。
- **用量统计按课程拆分**：`LLMTokenUsageLog` / 预留记录按 **学生 + 课程（`subject_id`）** 记账；**自然日**由该课程的 **`course_llm_configs.quota_timezone`** 决定（与管理员全局策略里的「额度统计时区」无关）。同一学生在不同课程下的已用 token **分别累计**，共用上述日上限数值。
- 若某课程尚未选择任何端点，教师打开或保存该课 LLM 配置时，系统会尝试从**另一门已配置且校验通过的课程**自动复制一份模板（无可用模板时端点列表为空）。
- 当某端点预设被停用或不再满足校验要求时，引用它的课程端点会被移除；在 PostgreSQL 上删除预设会级联删除课程端点关联行。
- **Token 限额**在批改流程中参与预检与预留；预算不足会导致任务失败或拒绝，**不是**「只要预算够就一定评成功」的单一条件。

## 3. 教师设置「每人最多提交次数」（`max_submissions`）

- 字段为 **`homeworks.max_submissions`**：`NULL` 表示不限制；正整数表示该作业下**每位学生**最多可产生多少次提交记录（`HomeworkAttempt`）。
- 教师下调该上限时，若会低于**任一学生已有提交次数**，接口会返回 **400**，避免数据与规则矛盾。
- 学生端接口会返回 **`submissions_remaining`**（剩余可提交次数），供前端展示与禁用提交按钮。

## 4. 与旧版本行为差异小结

| 项目 | 旧系统 / 未配置 LLM | 升级后 |
|------|---------------------|--------|
| 无 LLM | 无自动批改 | 仍可关闭自动批改，行为与纯人工一致 |
| 数据库 | 无新列/新表 | 启动迁移自动补齐 |
| 提交次数 | 一般不限制 | 可选 `max_submissions` 限制 |
| LLM 日用量 | （若曾按全局时区合并） | 按**课程 `quota_timezone`** 划分自然日，按**课程**累计学生输入 token |

若升级后仅出现「自动批改任务失败」类提示，请优先核对：**作业是否开启自动批改**、**课程 LLM 与端点**、**Worker 与网络**，而非怀疑升级未成功。
