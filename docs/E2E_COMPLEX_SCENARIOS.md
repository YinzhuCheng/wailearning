# Complex E2E Scenarios

本文档记录复杂 E2E 的设计原则、已实现覆盖、这轮测试中提炼出的风险点，以及下一轮仍值得投入的方向。

## 目标

复杂 E2E 不应只证明“按钮能点、接口能回”，而应验证：

1. 用户从真实 UI 操作开始。
2. 经过异步、并发、失败或跨角色状态传播后，系统最终仍收敛到正确状态。
3. UI 可见结果与后端 authoritative 状态一致。
4. 不出现重复、冲突或半完成记录。

## 已实现的复杂场景

当前已在 `frontend/e2e/e2e-scenario-resilience.spec.js` 中实现并跑通的重点场景包括：

### 并发编辑与重复提交

- 两个教师并发编辑同一作业，最终只能保留一个最终版本
- 作业创建首次失败后重试，只允许落一条记录
- 通知发布首次失败后重试，只允许落一条记录
- 两个教师陈旧花名册弹窗重复提交，只允许加一条课内记录

### 学生选课 / 退选并发

- 两个学生上下文并发自选同一选修课，只允许保留一次选课
- 两个学生上下文并发退选同一选修课，最终只允许收敛到未选课
- 学生停留在旧页面时被管理员调班，旧页面不得把学生重新选进不属于其班级的课程

### 申诉与通知并发

- 作业申诉并发提交时，只允许出现一条 pending
- 成绩申诉并发提交时，只允许出现一条 pending
- 双标签页同时打开同一通知，已读状态应幂等
- 双标签页同时“全部标为已读”，所有新通知都应变为已读

### 身份、上下文与缓存恢复

- 学生 fresh login 后直接深链路进入作业、通知、成绩页，系统应自动恢复课程上下文
- 本地 `selected_course` 缓存是脏值时，页面仍应恢复到真实课程上下文
- 管理员先把学生调走、再调回，学生课程访问应恢复

### 个人设置与失败恢复

- 个人资料首次保存失败后重试，最终只保留一次正确变更
- 两个陈旧设置页提交不同名字，最终应收敛到最后一次提交
- 修改密码先因旧密码错误失败，再次提交成功后，旧密码必须失效

### 默认值与部分配置

- 课程 LLM 配置在默认值缺失时应自动回填
- LLM 配置保存首次失败后再次保存，最终后端配置应一致

## 这轮测试真正暴露过的问题

以下问题并不是“理论风险”，而是这轮测试里实际暴露或逼近暴露边界的问题。

### 1. 深链路过早依赖 `selected_course`

风险：

- 学生从 fresh login 直接访问深链路页面时，本地没有课程上下文，前端会错误跳回 `/courses`。

启示：

- 学生页不应默认假设 localStorage 已经有课程缓存。

### 2. “先查再写”无法防并发

风险：

- 作业申诉、成绩申诉、选课、退选、通知已读这类接口，如果只做“先查询是否存在，再插入”，在并发下仍可能双写或抛 500。

启示：

- 复杂 E2E 非常适合筛出这种在单线程手工操作里不容易出现的问题。
- 真正重要的写路径应有数据库唯一约束或等价的最终一致性保护。

### 3. 课程、花名册、用户身份是强耦合链路

风险：

- 学生账号 `class_id`
- 花名册 `Student.class_id`
- 课程选课 `CourseEnrollment`
- 课程目录 / 可见课程

这几层只要有一层刷新不及时，就会出现“后端已变、UI未变”或“目录可见但实际未入课”这类复杂错位。

### 4. 通知读状态是典型的并发热点

风险：

- 同一用户在多标签页点通知、点“全部已读”，非常容易重复插入读状态。

启示：

- 看似“低风险的小功能”，反而因为交互高频、并发自然，会最早暴露数据幂等性问题。

### 5. 复杂测试本身也会脆

这轮有一个失败最终确认是测试代码问题而不是产品问题：

- 测试里课程名字符串被编码污染
- 定位器依赖展示文案
- 最终断言找错了元素

启示：

- 复杂 E2E 应尽量依赖稳定的 `data-testid`、结构定位和后端最终态校验。
- 展示文本适合辅助断言，不适合做唯一定位锚点。

## 当前仍然值得继续测试的方向

以下方向是下一轮最值得继续打的。

### 1. 学生改学号 / 改用户名后的全链路一致性

为什么重要：

- 系统中用户登录名与花名册学号强绑定。
- 一旦管理员改用户名，课程可见性、作业提交、成绩归属、家长码、通知 target 都可能受到影响。

建议检查：

- 旧 token 行为
- 新用户名重新登录后的课程恢复
- 花名册 `student_no` 是否同步
- 作业 / 成绩 / 通知是否仍绑定到正确学生

### 2. LLM 执行期失败矩阵

当前已覆盖：

- 配置保存失败
- 默认值回填

仍未系统覆盖：

- timeout
- 429 / quota exhausted
- 5xx
- 非 JSON 返回
- 空响应
- 多 endpoint fallback
- 后台任务卡在 queued / running
- 轮询恢复与二次登录恢复

### 3. 长链路异步恢复

重点不是“写接口失败”，而是：

- 写接口成功，但后续刷新失败
- 后台任务成功，但前端轮询失败
- 用户刷新页面或重新登录后，能否重新看到 authoritative 状态

### 4. 冷启动与半初始化数据

建议继续测试：

- 新库首次启动
- migration 后只有部分新字段
- 配置对象缺少 optional 字段
- 老记录没有默认值但新页面已开始依赖这些值

### 5. 更真实的多角色并发

比如：

- 管理员批量调班同时教师打开花名册
- 教师改课程同时学生自选课
- 教师处理申诉同时学生刷新提交页
- 家长端、学生端、教师端同时读取同一学生的关联数据

## 复杂 E2E 的落地准则

一个复杂场景如果要值得保留，至少应同时满足：

- 有明确的 UI 起点
- 有可见结果或明确失败反馈
- 有后端最终态校验
- 能验证“没有重复、没有半完成、没有冲突残留”

如果一个场景只验证单个接口返回值或单个按钮可点击，它更适合写成 `pytest` 或普通回归测试，而不是复杂 E2E。
